BubbleBreak Problem Solved 요약
작성일: 2026-05-13

목표:
Cloudflare에 배포한 BubbleBreak에서 방 생성이 안 되고, 카카오톡/DM 링크 미리보기가 이상하게 뜨는 문제를 해결했다.


1. 문제: "방만들고 입장하기" 클릭 시 Failed to fetch

증상:
- SetupScreen에서 "방만들고 입장하기" 버튼을 누르면 화면 아래에 "Failed to fetch"가 표시됐다.
- Chrome DevTools Network 탭에서 /rooms 요청이 실패했다.

원인:
- Cloudflare에 배포된 FE가 Cloudflare BE를 호출하지 않고 로컬 개발 서버 주소를 호출하고 있었다.
- 배포된 JS 안에 아래 주소가 들어가 있었다.

  http://127.0.0.1:8787

- 이 주소는 사용자의 컴퓨터/핸드폰 자기 자신을 뜻한다.
- 그래서 Cloudflare 사이트를 열어도 API 요청은 Cloudflare Worker가 아니라 사용자 기기의 로컬 서버로 가려고 했다.

해결:
- 배포 환경에서는 API 주소를 비워서 같은 Cloudflare 도메인으로 요청하도록 수정했다.
- 이제 FE는 /rooms로 요청하고, 같은 도메인의 Cloudflare Worker가 처리한다.

수정 파일:
- apps/web/src/config/env.ts
- apps/worker/package.json


2. 문제: Cloudflare Dashboard에 Requests와 Errors가 같이 증가

증상:
- Cloudflare Dashboard에서 요청이 들어오지만 전부 Errors로 찍혔다.
- 배포 URL의 /health를 확인했을 때 500 error code: 1101이 발생했다.

원인:
- Cloudflare Worker는 배포되어 있었지만, 연결된 원격 D1 DB에 테이블이 없었다.
- users, rooms, room_participants, room_events, room_sessions 같은 테이블이 생성되지 않은 상태였다.
- 즉, 서버는 있었지만 서버가 사용할 DB 구조가 준비되지 않았다.

해결:
- 원격 D1 DB에 migration을 적용했다.

실행한 명령:

  yarn --cwd apps/worker wrangler d1 migrations apply bubblebreak-db --remote

결과:
- /health 요청이 200 OK로 바뀌었다.
- POST /rooms 요청이 201 Created로 성공했다.


3. 문제: 로컬 Cloudflare Worker dev 서버가 켜지지 않음

증상:
- yarn --cwd apps/worker dev 실행 시 Worker가 시작되지 않았다.

원인:
- wrangler.jsonc의 compatibility_date가 현재 로컬 wrangler/workerd가 지원하는 날짜보다 높았다.

오류 내용:

  This Worker requires compatibility date "2026-04-27",
  but the newest date supported by this server binary is "2026-04-08".

해결:
- 로컬에서 실행 가능한 날짜로 compatibility_date를 조정했다.

수정 파일:
- apps/worker/wrangler.jsonc


4. 문제: 카톡/DM 링크 미리보기에 "Mobile-first UI/UX Design"이 뜸

증상:
- BubbleBreak 링크를 카카오톡이나 DM으로 보내면 미리보기 제목이 "Mobile-first UI/UX Design"으로 표시됐다.

원인:
- HTML의 title이 예전 템플릿 값으로 남아 있었다.

기존 값:

  <title>Mobile-first UI/UX Design</title>

해결:
- title과 Open Graph 메타태그를 BubbleBreak용으로 수정했다.
- 링크 미리보기용 로고 이미지도 추가했다.

변경된 제목:

  아이스브레이킹은 BubbleBreak로

추가한 미리보기 이미지:

  /bubblebreak-logo.png

수정 파일:
- apps/web/index.html
- apps/web/public/bubblebreak-logo.png


5. 문제: 카톡 미리보기가 수정 후에도 바로 안 바뀜

증상:
- HTML을 수정하고 배포했는데도 카톡에서는 예전 미리보기가 계속 보였다.

원인:
- 카톡이 같은 URL의 미리보기 결과를 캐시하고 있었다.
- 사이트는 이미 바뀌었지만, 카톡이 예전 preview 데이터를 재사용했다.

해결:
- URL 뒤에 query string을 붙여 새 링크처럼 인식하게 했다.

예시:

  https://bubblebreak-demo.jinung344.workers.dev/?v=2

설명:
- ?v=2는 앱 동작에는 영향을 주지 않는다.
- 하지만 카톡은 다른 URL로 보고 미리보기를 다시 가져온다.


6. 최종 정상 흐름

현재 요청 흐름:

  사용자
  -> Cloudflare에 배포된 FE 접속
  -> FE가 /rooms 요청
  -> 같은 Cloudflare Worker BE가 처리
  -> Cloudflare D1 DB에 방 정보 저장
  -> 방 생성 성공


7. 최종 확인 결과

정상 확인:

  GET /health
  -> 200 OK

  POST /rooms
  -> 201 Created

링크 미리보기:

  제목:
  아이스브레이킹은 BubbleBreak로

  이미지:
  BubbleBreak 로고


8. 한 줄 회고

이번 문제는 단순한 fetch 오류가 아니라, 배포 환경에서 FE API 주소, Cloudflare D1 migration, HTML preview metadata, 카톡 캐시가 각각 얽혀 있었던 문제였다. 하나씩 분리해서 확인하면서 최종적으로 FE-BE-DB-링크 공유 흐름을 정상화했다.
